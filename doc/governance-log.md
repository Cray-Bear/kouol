1. **三权分离**：议（设计）→ 行（实施）→ 审（验证）

### 费曼输出标准（第三个思想体系）

来源：[feynman-skill](https://github.com/alchaincyf/feynman-skill)

**核心公式：** 具体例子开场 + 不用术语 + 主动标记不确定性 + 演示替代论证

融入方式：不独立成章节，直接嵌入每个角色的输出格式中作为质量注释。

| 角色 | 融入位置 | 关键原则 |
|------|---------|---------|
| Strategist | 想法格式 | 不用术语、从具体场景开始、诚实标记不确定性 |
| Architect | 技术方案格式 | 命名≠理解、先给场景再抽象 |
| PM | 需求描述格式 | 货物崇拜检测、用户能看懂吗？ |
| Developer | 日志记录格式 | 命名表达意图、演示>论证、标记不确定性 |
| Designer | 日志条目格式 | 具象化思考、不用设计术语解释决策 |
| Tester | Bug 列表格式 | O 型环范式：10 秒可复现演示 |
| Reviewer | 改进建议格式 | 反问句替代感叹句、反自欺 |
| Auditor | 问题发现格式 | 货物崇拜检测、直接验证、反身份固化 |

## 2026-07-19 — 六权模型：PM 合并给战略者，运维合并给开发者

### 背景
- 用户提出：当前项目是否需要独立的运维和产品角色
- 分析结论：单用户教学项目中，PM 和运维都是冗余角色

### 变更内容

| 变更 | 说明 |
|------|------|
| 📋 PM → 🎯 Strategist | 需求定义、验收标准、任务拆解全部由战略者负责 |
| 运维 → 💻 Developer | 部署验证、本地测试、线上检查由开发者负责 |
| 七权 → 六权 | 策+产品 / 议 / 行+运维 / 审 / 督 |
| 8 Agent → 7 Agent | 删除 kouol-pm Skill |

### 分析依据

**PM 为什么冗余：**
- "为什么做"（战略者）和"做什么"（PM）是同一件事的两面
- 验收标准定义 → 测试者基于它验证，谁来定义不重要
- 单用户项目，用户直接裁决，不需要"用户代言"中间层

**运维为什么冗余：**
- GitHub Pages 自动部署，无独立服务器/数据库
- 部署验证是交付的一部分（"交付"="确认能跑"）
- 教学项目没有独立的运维技术基础

## 2026-07-19 — 影响范围决定流程重量（三级自治框架）

### 背景
- 设计师巡检发现问题时，原先全部转为 Idea 走流程
- 用户指出：应该根据影响范围区分处理——只影响自己的直接改，影响下游的沟通后改，影响已动工的全流程

### 核心原则

**流程重量 = 协调风险 = 影响范围 × 下游活跃度**

| 情况 | 判断 | 处理 |
|------|------|------|
| 只影响自己 | 下游没动这个产出 | 直接改，记录即可 |
| 影响下游但下游没动 | 产出已交给下游但未动工 | 和战略者沟通后改 |
| 影响已动的下游 | 下游已在讨论/开发/已上线 | 提 Idea 走全流程 |

### 第一性原理分析

**隐含假设识别：**
- 旧假设："所有变更都应该走 Idea → Requirement 流程"
- 推翻：流程的目的是防止协调失败，不是为了展示"我们很规范"。如果协调风险为零，流程本身就是浪费。

**公理化推导：**
```
公理：流程重量 = 协调风险
      协调风险 = 影响范围 × 下游活跃度

当 协调风险 ≈ 0 → 流程重量 ≈ 0（直接改）
当 协调风险 > 0 → 流程重量 > 0（沟通或全流程）
```

**破界创新：**
- 破：变更必须走流程
- 立：变更的流程重量由影响范围决定
- 见：**自治 = 影响范围只在自己时直接行动**

### 对"七权协作"的补充

协作不是所有事都一起讨论，而是**知道什么时候不需要协作**。

| Agent | 可以直接改的 | 需要沟通的 | 需要走流程的 |
|-------|------------|-----------|------------|
| 🎯 Strategist | 想法池中的 I-xxx | 未进入议的 R-xxx 初稿 | 已批准的 R-xxx |
| 🎨 Designer | 未交给下游的设计稿 | 已交给下游但未动工 | 已开发或已上线的页面 |
| 🏗️ Architect | 未交给 Developer 的技术方案 | Developer 还没开始编码的方案 | Developer 正在编码的方案 |
| 💻 Developer | 不影响接口的重构 | 内部实现优化 | 影响其他模块的变更 |
| 🧪 Tester | 测试用例措辞/格式 | 验收标准的理解 | 验收标准本身 |

### 通用规则

1. **直接改也要记录** — discussion.md 或 log.md 说明改了什么、为什么
2. **不确定就升级** — 拿不准影响范围时，走更重的流程
3. **所有 Agent 同理** — 这个原则适用于每个角色
