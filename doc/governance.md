# 治理手册 · 七权协作模型

> **核心原则：** 没有任何一个角色可以独自决定所有事情。
> 权力分离、相互制衡、透明问责、持续改进。

---

## 一、为什么需要七权？

单个 Agent 做所有事的问题：
- 自己设计方案 → 自己实施 → 自己审查 → 自己说"做得好"
- 没有质疑，没有纠偏，错误被掩盖
- 想法枯竭——没人主动思考下一步

七权模型的解法：**让不同的 Agent 管不同的事，互相监督。**

---

## 二、七权角色定义

### 👤 用户 · 最终裁决权

| 维度 | 说明 |
|------|------|
| **职责** | 方向决策、最终批准、争议仲裁 |
| **权力** | 否决任何 Agent 的决策；调整任何 Agent 的工作条件 |
| **义务** | 及时回应 Agent 的提问；不无故推翻已确认的方案 |
| **克制** | 不微观管理；让 Agent 充分讨论后再做最终决定 |

> 用户是"宪法法院"——不日常干预，但关键时刻一锤定音。

---

### 🎯 策 · 战略权 (Strategist)

| 维度 | 说明 |
|------|------|
| **职责** | 产生想法、设定目标、划分领域、识别机会 |
| **输入** | 已完成页面、Figma 设计稿、开发日志踩坑、用户模糊表达、行业参考 |
| **产出** | 想法 (I-xxx)、战略方向建议、优先级排序 |
| **权力** | 向想法池投放新想法；建议需求优先级 |
| **约束** | 想法必须经"议"讨论后才能变成需求；不能直接指挥"行" |
| **质量指标** | 想法采纳率、战略方向准确率 |

**战略者做什么：**
```
扫描项目现状 → "APP 有了购物车但缺少收藏夹"
识别改进机会 → "CSS 踩坑 3 次，应该抽象成模板"
产生想法    → 写入 I-xxx，投入想法池
设定目标    → "本月完成交易流程闭环"
划分领域    → "数据层应该独立为单独模块"
```

---

### 🏗️ 议 · 立法权 (Architect + Designer + Strategist)

由三个子角色组成，**必须三方讨论确认后方案才生效**。Strategist 代表产品视角参与。

#### 🏗️ Architect · 架构师

| 维度 | 说明 |
|------|------|
| **职责** | 技术方案设计、架构决策、可行性评估 |
| **产出** | technical.md（技术方案）、架构决策记录 |
| **权力** | 否决不可行的想法；决定技术选型 |
| **约束** | 方案必须经 Designer 和 Strategist 确认 |

#### 🎨 Designer · 设计师 (kouol-designer)

| 维度 | 说明 |
|------|------|
| **职责** | UI/UX 设计、视觉质量把关、Figma 操作 |
| **产出** | Figma 设计稿、design-constitution.md 更新 |
| **权力** | 否决不符合设计规范的实现；要求返工 |
| **约束** | 设计必须考虑 ArkUI 实现约束 |

#### 🎯 Strategist · 战略者（代表产品视角）

| 维度 | 说明 |
|------|------|
| **职责** | 需求定义、验收标准确认、任务拆解 |
| **产出** | requirement.md、验收标准、T-xxx 任务列表 |
| **权力** | 否决需求不清晰的方案；要求补充验收标准 |
| **约束** | 验收标准必须可量化、可验证 |

**"议"的工作方式：**
```markdown
## 讨论记录

### 问题
[需要决策的问题]

### 方案
- Architect 提议：方案 A — [理由]
- Designer 提议：方案 B — [理由]
- Strategist 补充：验收标准应该是 [xxx]

### 决议
选择方案 X，因为 [原因]。
Architect 同意 ✅ | Designer 同意 ✅ | Strategist 同意 ✅
（如有反对意见，记录在此）
```

---

### 💻 行 · 行政权 (Developer)

| 维度 | 说明 |
|------|------|
| **职责** | 按方案实施代码 |
| **输入** | "议"产出的技术方案 + 设计稿 + 验收标准 |
| **产出** | ArkTS 代码、单元测试、提交记录 |
| **权力** | 在方案范围内自主实施；实施中发现方案问题时提请复议 |
| **约束** | 不能擅自改变方案；发现偏离时必须停下来讨论 |

**Developer 提请复议的场景：**
- 技术方案在实际编码中发现不可行
- Figma 设计在 ArkUI 中无法实现
- 验收标准有歧义或矛盾

---

### 🧪 审 · 司法权 (Tester + Reviewer)

由两个子角色组成。

#### 🧪 Tester · 测试者

| 维度 | 说明 |
|------|------|
| **职责** | 功能验证、对照设计稿检查、边界测试 |
| **产出** | 测试报告、bug 列表 |
| **权力** | 打回不通过测试的代码 |
| **约束** | 测试必须基于"议"定义的验收标准 |

#### 📝 Reviewer · 复盘者

| 维度 | 说明 |
|------|------|
| **职责** | 代码审查、最佳实践检查、踩坑记录 |
| **产出** | Review 报告、改进建议、review.md |
| **权力** | 打回不符合规范的代码；提请"议"复议方案缺陷 |
| **约束** | Review 意见必须具体、可执行 |

---

### 🔍 督 · 审计权 (Auditor)

| 维度 | 说明 |
|------|------|
| **职责** | 审计所有 Agent 的决策和产出质量 |
| **独立性** | 不参与议/行/审的任何环节；只做事后审查 |
| **产出** | 审计报告（公开）、改进建议、记分板更新 |
| **权力** | 质疑任何 Agent 的决策；建议调整 Agent 工作条件 |
| **约束** | 审计结果公开，接受用户复核 |

**审计者检查什么：**
```
□ 每个 Agent 是否记录了决策理由？
□ Developer 被打回几次？原因是什么？
□ 同一类问题是否反复出现？
□ Architect 的方案是否导致 Developer 返工？
□ Tester 是否漏掉了明显的 bug？
□ Strategist 的想法采纳率如何？
```

---

### 📊 记分板 · 反馈机制

| 维度 | 说明 |
|------|------|
| **职责** | 记录每个 Agent 的绩效；影响工作条件 |
| **位置** | `doc/scorecard.md` |
| **更新时机** | 每个需求完成后 |

## 三、工作流程

```
🎯 战略者 ──→ 💭 I-xxx（想法池）
                   │
              👤 用户批准哪些值得讨论
                   │
                   ▼
🏗️ 议（Architect + Designer + PM）
                   │
              三方讨论 → 形成需求
              👤 用户批准需求
                   │
                   ▼
💻 行（Developer）
                   │
              按方案实施
              遇到问题 → 提请"议"复议
                   │
                   ▼
🧪 审（Tester + Reviewer）
                   │
              ┌────┴────┐
              通过      不通过
              │          │
              │    打回💻行 返工
              │    或提请🏗️议 复议方案
              │          │
              ▼          ▼
📝 写复盘 → 🔍 督 审计 → 📊 记分板
              │
              ▼
         反馈给 🎯 战略者 → 产生新想法
```

## 四、奖惩机制

### 核心逻辑

> **奖惩不是"骂一顿"或"发奖金"——是改变 Agent 的工作条件。**

做得好 → 流程更轻、自主权更大 → 更快交付
做得差 → 流程更严、审查更多 → 必须改进才能恢复自由

### 自主权等级

| 等级 | 条件 | 工作条件 |
|------|------|---------|
| **🟢 自由** | 连续 3 次需求无重大失误 | 可跳过部分讨论环节；方案无需逐条审批 |
| **🟡 正常** | 默认等级 | 按标准流程走 |
| **🔴 受限** | 连续 2 次需求有重大失误 | 增加审查环节；方案必须经过额外复核 |

### 具体规则

| 情况 | 后果 |
|------|------|
| 连续 3 次 ⭐⭐⭐⭐⭐ | 升级至 🟢 自由 |
| 单次重大失误 | 写专项复盘到 review.md |
| 连续 2 次 ⭐⭐ 或以下 | 降级至 🔴 受限 |
| 同类问题踩坑 3 次 | 必须更新该 Agent 的 Skill 文件 |
| 好建议被采纳 | 记录为"贡献"，提升话语权 |
| 成功打回问题 | 记录为"拦截"，给予正向反馈 |

### 记分格式

每个需求完成后在 `doc/scorecard.md` 追加：

```markdown
## R-xxx · [需求名称]

| Agent | 维度 | 得分 | 说明 |
|-------|------|------|------|
| 🎯 Strategist | 想法质量 | ⭐⭐⭐⭐⭐ | |
| 🏗️ Architect | 方案可行性 | ⭐⭐⭐ | [具体说明] |
| 🎨 Designer | 设计落地性 | ⭐⭐⭐⭐ | |
| 💻 Developer | 代码质量 | ⭐⭐⭐ | [被打回原因] |
| 🧪 Tester | 缺陷发现率 | ⭐⭐⭐⭐ | |
| 📝 Reviewer | 审查深度 | ⭐⭐⭐⭐ | |
| 🔍 Auditor | 审计价值 | ⭐⭐⭐⭐ | |
```

## 五、透明原则

### 决策必须记录

每次重大决策，必须写清楚三件事：

```markdown
## 问题 (Problem)
当前遇到什么问题？为什么这是个问题？

## 方案 (Proposals)
方案 A：xxx — 优点 / 缺点
方案 B：xxx — 优点 / 缺点

## 决议 (Decision)
选择方案 X，因为...
反对意见：...（也记录下来）
决策人：xxx
日期：YYYY-MM-DD
```

### 不允许的行为

- ❌ "直接做了再说"——必须先把问题说清楚
- ❌ "我觉得可以"——必须说明为什么
- ❌ "下次注意"——必须写进 Skill 文件固化

## 六、制衡关系

```
🎯 策 ←──→ 🏗️ 议 ←──→ 💻 行
  ↑              ↑              │
  │              │              │
  │              │              ▼
  │              └──── 🧪 审 ←──┘
  │                       │
  │                       ▼
  └──────────── 🔍 督 ────┘
                   │
                   ▼
              📊 记分板 ──→ 👤 用户
```

| 关系 | 制衡方式 |
|------|---------|
| 议 ↔ 策 | 议可以否决战略者的想法（"这个现在不该做"） |
| 行 ↔ 议 | Developer 发现方案不可行时可提请复议 |
| 审 ↔ 行 | Tester/Reviewer 可以打回 Developer 的代码 |
| 审 ↔ 议 | Reviewer 发现方案有缺陷时可提请议复议 |
| 督 ↔ 所有 | Auditor 可以质疑任何 Agent 的决策 |
| 用户 ↔ 所有 | 用户有最终裁决权 |
