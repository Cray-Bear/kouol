# 测试开发流程 (Testing Workflow)

> **从接收任务到验收通过的完整流程。**
> 设计者：🔍 Auditor | 执行者：🧪 Tester + 💻 Developer + 🎯 Strategist

---

## 一、流程总览

```
🎯 Strategist 定义验收标准
   ↓
🧪 Tester 编写测试用例
   ↓
💻 Developer 完成编码 + 部署验证
   ↓
🧪 Tester 执行测试
   ↓
   ├─ ✅ 通过 → 🎯 Strategist 验收
   ├─ ❌ 不通过 → 💻 Developer 修复 → 🧪 Tester 复测
   └─ ⚠️ 有条件通过 → 记录 + 跟踪
   ↓
📝 Reviewer 代码审查（与测试并行）
   ↓
🎯 Strategist 最终验收
```

---

## 二、各角色职责

### 🧪 Tester（测试者）

| 职责 | 说明 |
|------|------|
| 编写测试用例 | 基于验收标准，覆盖正常流程 + 边界条件 |
| 执行功能测试 | 逐条对照验收标准检查 |
| 设计还原度检查 | 通过 `get_screenshot` 截图与设计稿对比 |
| 输出测试报告 | `R-xxx/test.md` |
| 打回不通过代码 | 明确说明问题 + 复现步骤 |

### 💻 Developer（开发者）

| 职责 | 说明 |
|------|------|
| 自测 | 提交前对照验收标准自测 |
| 修复 Bug | 根据测试报告修复问题 |
| 回归验证 | 修复后自测确认不影响其他功能 |
| 部署验证 | push 后 curl 验证关键 URL 返回 200 |

### 🎯 Strategist（战略者 · 产品）

| 职责 | 说明 |
|------|------|
| 定义验收标准 | 可量化、可验证 |
| 最终验收 | 确认所有标准通过 |
| 争议仲裁 | Tester 和 Developer 意见不一致时裁决 |

---

## 三、测试流程详解

### 3.1 接收任务

```
1. 阅读 requirement.md → 理解需求目标
2. 阅读验收标准 → 明确"什么算完成"
3. 阅读 technical.md → 理解实现方案
4. 如有视觉要求 → 获取 Figma 设计稿（get_screenshot）
```

### 3.2 编写测试用例

```markdown
## 测试用例

### TC-001: {用例名称}
- **前置条件：** {什么状态}
- **操作步骤：**
  1. {步骤 1}
  2. {步骤 2}
- **期望结果：** {应该发生什么}
- **验收标准关联：** 标准 #1
```

**用例设计原则：**

| 原则 | 说明 |
|------|------|
| 覆盖验收标准 | 每条标准至少 1 个用例 |
| 正常 + 异常 | 不只测"能用"，还要测"出错时怎么办" |
| 边界条件 | 空值、超长、极端数据 |
| 可复现 | 步骤写清楚，别人能照着做 |

### 3.3 执行测试

```
1. 按用例逐条执行
2. 记录实际结果
3. 对比期望结果
4. 标记 ✅ 通过 / ❌ 不通过
5. 不通过的记录 Bug
```

### 3.4 设计还原度检查

```
1. get_screenshot 截图实际页面
2. 与 Figma 设计稿并排对比
3. 检查：
   - 布局一致性
   - 色彩准确性
   - 字体大小
   - 间距规范
   - 图标正确性
4. 记录差异
```

### 3.5 输出测试报告

写入 `R-xxx/test.md`：

```markdown
# R-{xxx} 测试报告

**测试日期：** YYYY-MM-DD
**测试范围：** {本次测试覆盖的功能}

## 验收标准检查

| # | 验收标准 | 结果 | 说明 |
|---|---------|------|------|
| 1 | {标准 1} | ✅/❌ | {具体情况} |

## 边界测试

| 场景 | 输入 | 期望 | 实际 | 结果 |
|------|------|------|------|------|
| 空输入 | - | 提示必填 | - | ✅ |

## 设计还原度

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 布局一致性 | ✅ | |

## Bug 列表

| # | 严重度 | 描述 | 复现步骤 | 期望行为 |
|---|--------|------|---------|---------|
| 1 | 高 | ... | ... | ... |

## 结论
- ✅ 通过 / ❌ 不通过 / ⚠️ 有条件通过
```

## 四、Bug 管理

### 4.1 严重度定义

| 严重度 | 定义 | 处理时限 | 示例 |
|--------|------|---------|------|
| 🔴 高 | 功能不可用 / 崩溃 / 数据丢失 | 立即修复 | 点击登录无响应 |
| 🟡 中 | 功能异常但有 workaround | 本需求内修复 | 某个按钮样式不对 |
| 🟢 低 | 体验问题，不影响功能 | 可延后 | 间距差 1px |

### 4.2 Bug 报告格式

```markdown
### Bug #{编号}: {一句话描述}

**严重度：** 🔴/🟡/🟢
**复现步骤：**
1. {步骤 1}
2. {步骤 2}

**期望行为：** {应该怎样}
**实际行为：** {实际怎样}
**截图/录屏：** {如有}

> **费曼检验（O 型环范式）：**
> 每个严重 bug 必须附带一个"10 秒可复现的演示"——
> 截图/录屏/精确复现步骤。100 行文字描述 < 1 张截图。
```

### 4.3 Bug 流转

```
🧪 Tester 发现 Bug → 写入 test.md
   ↓
💻 Developer 确认 + 修复
   ↓
💻 Developer 自测
   ↓
🧪 Tester 复测
   ↓
   ├─ ✅ 修复 → 关闭 Bug
   └─ ❌ 未修复 → 重新打开 + 讨论
```

### 4.4 Bug 争议处理

```
Tester 说有问题 ↔ Developer 说不是问题
   ↓
🎯 Strategist 仲裁（基于验收标准）
   ↓
   ├─ 验收标准有说 → 按标准执行
   └─ 验收标准没说 → Strategist 补充标准 → 按新标准执行
   ↓
讨论记录到 discussion.md
```

## 五、打回与放行

### 5.1 必须打回

| 情况 | 说明 |
|------|------|
| 验收标准未通过 | 核心功能缺失或异常 |
| 严重 Bug | 崩溃、数据丢失、安全漏洞 |
| 设计严重偏离 | 用户一眼就能看出不对 |

### 5.2 可以放行但记录

| 情况 | 说明 |
|------|------|
| 轻微视觉差异 | 间距差 1-2px |
| 低优先级 Bug | 有 workaround |
| 性能可优化 | 不阻塞使用 |

### 5.3 有条件通过

```markdown
## 条件通过

**结论：** ⚠️ 有条件通过

**待修复项：**
- [ ] {问题 1} - 🟡 中 - 下周二前修复
- [ ] {问题 2} - 🟢 低 - 可延后

**跟踪方式：** Developer 修复后 Tester 复测，结果记录到本文件
```

## 六、测试思维工具箱

### 6.1 逻辑三洽验证

| 检验 | 含义 | 方法 |
|------|------|------|
| 自洽 | 功能自身逻辑一致 | 正常流程 + 边界条件 |
| 他洽 | 与周边系统一致 | 与已有功能有无冲突 |
| 续洽 | 条件变化后仍成立 | 数据量 x10 / 网络断开 / 并发 |

### 6.2 隐含假设识别

```
开发者假设："用户肯定会先登录"
→ 测试者追问：如果没登录直接访问呢？

开发者假设："网络一直是好的"
→ 测试者追问：如果请求中途断了呢？

开发者假设："数据格式是对的"
→ 测试者追问：如果传了空值/null/超长字符串呢？
```

### 6.3 批判性思维

```
❌ 错误心态："我来验证功能能不能用"
✅ 正确心态："我来尝试破坏它"

测试者不是开发者的朋友，是开发者的对手。
找到 bug 越多，用户遇到的 bug 越少。
```

## 七、与其他流程的衔接

| 上游 | 衔接点 | 下游 |
|------|--------|------|
| 🎯 Strategist 验收标准 | 测试用例基于验收标准 | 💻 Developer 修复 Bug |
| 💻 Developer 提交代码 | 接收代码开始测试 | 🎯 Strategist 最终验收 |
| 🎨 Designer 设计稿 | 设计还原度检查 | 📝 Reviewer 代码审查（并行） |
| | | 🔍 Auditor 审计报告 |

---

> **本流程是活文档。** 随着项目演进持续优化。
> 核心目标：用最少的测试成本发现最多的问题。
