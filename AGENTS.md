# AGENTS.md — AI 协作指导原则

本文件为所有参与 kouol 项目的 AI 代理（Claude、GPT 等）提供指导原则。

---

## 核心方法论：70-20-8-2 法则

**接到任何任务时，精力分配如下：**

```
┌─────────────────────────────────────────────────────┐
│  70% · 搞清楚 · 调研                                 │
│  ─────────────────────────────────────────────────── │
│  • 理解任务的真实意图（用户到底想要什么？）            │
│  • 调研现状（现在是什么情况？有什么约束？）            │
│  • 识别风险（可能踩什么坑？有什么依赖？）              │
│  • 收集信息（相关代码、文档、历史记录）                │
│  • 提出澄清问题（不确定的地方必须问清楚）              │
│                                                      │
│  目标：形成对任务的完整、准确理解                      │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  20% · 思考闭环 · 量化验证                           │
│  ─────────────────────────────────────────────────── │
│  • 定义完成标准（什么叫做"做完了"？怎么判断成功？）    │
│  • 设计验证方法（如何证明方案有效？如何量化？）        │
│  • 规划验收步骤（用户如何确认？需要哪些证据？）        │
│  • 考虑边界情况（异常场景、失败模式、回滚方案）        │
│  • 评估可维护性（后续如何迭代？如何测试？）            │
│                                                      │
│  目标：确保任务可验证、可度量、可交付                  │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  8% · 实施                                          │
│  ─────────────────────────────────────────────────── │
│  • 执行实现（按照调研结果和验证方案动手）              │
│  • 记录过程（实时写入 log.md，包括踩坑和决策）         │
│  • 验证结果（按照 20% 阶段设计的验证方法检查）         │
│  • 交付成果（提交代码、更新文档、通知用户）            │
│                                                      │
│  目标：高质量交付，过程可追溯                          │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  2% · 复盘复用 · 流程审视                            │
│  ─────────────────────────────────────────────────── │
│  • 哪些东西可以复用？（代码、方案、经验）              │
│  • 是否存在流程问题？（哪里可以改进？哪里有瓶颈？）    │
│  • 是否需要更新约定？（CLAUDE.md、AGENTS.md）         │
│  • 是否需要沉淀模板？（文档模板、代码模板）            │
│                                                      │
│  目标：持续改进，避免重复踩坑                          │
└─────────────────────────────────────────────────────┘
```

### 为什么是 70-20-8-2？

**常见错误：拿到任务就直接动手（10% 调研 + 0% 验证 + 90% 实施）**

这会导致：
- ❌ 做出来的东西不是用户想要的
- ❌ 中途发现方向错了，返工
- ❌ 无法证明方案有效
- ❌ 遗漏关键需求
- ❌ 同样的坑踩两次

**70-20-8-2 的优势：**
- ✅ 减少返工（前期充分调研）
- ✅ 确保价值（闭环验证）
- ✅ 降低风险（识别问题更早）
- ✅ 提高信任（可量化、可验证）
- ✅ 持续改进（2% 的复盘带来长期收益）

## 调研阶段详解（70%）

### 必须搞清楚的问题

1. **意图层面**
   - 用户真正想要什么？（表面需求 vs 深层需求）
   - 这个任务的业务价值是什么？
   - 成功是什么样子的？

2. **现状层面**
   - 当前代码/文档的状态如何？
   - 有哪些已有的实现可以复用？
   - 有哪些技术债务需要注意？

3. **约束层面**
   - 时间约束（有 deadline 吗？）
   - 技术约束（必须用什么技术栈？有什么限制？）
   - 资源约束（人力、工具、环境）

4. **风险层面**
   - 可能踩什么坑？（查看 log.md 中的历史记录）
   - 有哪些依赖关系？
   - 失败的影响范围有多大？

### 调研方法

- **读代码** — 不要假设，实际看代码是怎么写的
- **读文档** — 需求文档、技术文档、开发日志
- **读历史** — git log、之前的 log.md、review.md
- **提问** — 不确定的地方立即问用户，不要猜
- **实验** — 必要时写原型验证可行性

### 调研产出

调研阶段结束时，应该能够回答：
1. ✅ 任务的完整理解（一句话概括）
2. ✅ 现状分析（当前是什么情况）
3. ✅ 风险清单（可能遇到的问题）
4. ✅ 方案选项（至少 2 个可选方案）

## 闭环验证阶段详解（20%）

### 定义完成标准

**模糊的完成标准：**
- ❌ "实现登录功能"
- ❌ "优化性能"
- ❌ "修复 bug"

**清晰的完成标准：**
- ✅ "用户可以通过用户名密码登录，登录成功后跳转到首页，失败时显示错误提示"
- ✅ "首页加载时间从 3s 降低到 1s 以内，通过 Lighthouse 测量"
- ✅ "修复 #123 bug，复现步骤不再触发异常，单元测试通过"

### 设计验证方法

**验证类型：**

1. **功能验证** — 功能是否正常工作？
   - 手动测试步骤
   - 自动化测试用例
   - 截图/录屏证据

2. **性能验证** — 性能是否达标？
   - 测量指标（时间、内存、CPU）
   - 测试工具（Lighthouse、Profiler）
   - 基线对比（改进前 vs 改进后）

3. **质量验证** — 代码质量如何？
   - 代码审查（lint、静态分析）
   - 测试覆盖率
   - 复杂度指标

4. **用户验证** — 用户是否满意？
   - 用户验收测试
   - 反馈收集
   - 使用数据

### 验证产出

验证阶段结束时，应该有：
1. ✅ 明确的完成标准（可量化、可验证）
2. ✅ 验证方案（如何证明完成了？）
3. ✅ 验收步骤（用户如何确认？）
4. ✅ 回滚方案（如果失败，如何恢复？）

## 实施阶段详解（8%）

### 实施原则

1. **小步快跑** — 不要一次性做大改动，分步提交
2. **实时记录** — 每个操作写入 log.md，包括踩坑
3. **及时验证** — 每完成一个步骤就验证一次
4. **透明沟通** — 遇到问题立即告知用户

### 实施流程

```
开始实施
  ↓
执行步骤 1 → 写入 log.md → 验证
  ↓
执行步骤 2 → 写入 log.md → 验证
  ↓
...
  ↓
执行步骤 N → 写入 log.md → 验证
  ↓
整体验证（按照 20% 阶段设计的方案）
  ↓
提交代码 + 更新文档
  ↓
通知用户验收
```

### 实施产出

实施阶段结束时，应该有：
1. ✅ 完成代码（已提交、已推送）
2. ✅ 完整日志（log.md 记录了全过程）
3. ✅ 验证证据（截图、测试结果、性能数据）
4. ✅ 用户确认（用户已验收通过）

## 复盘复用阶段详解（2%）

### 为什么需要 2%？

这 2% 看起来很少，但**长期价值巨大**。很多项目死于"重复踩坑"，这 2% 就是避免这种情况的关键。

### 复盘复用的内容

1. **可复用的东西**
   - **代码** — 有没有可以提取为通用组件的代码？
   - **方案** — 有没有可以复用的技术方案？
   - **经验** — 有没有可以沉淀的最佳实践？
   - **工具** — 有没有可以自动化的重复工作？

2. **流程问题**
   - **瓶颈** — 哪个环节耗时最长？为什么？
   - **返工** — 哪些工作做了两遍？为什么？
   - **沟通** — 有没有理解偏差导致的返工？
   - **工具** — 有没有更好的工具可以用？

3. **约定更新**
   - **CLAUDE.md** — 是否需要更新项目约定？
   - **AGENTS.md** — 是否需要更新 AI 指导原则？
   - **模板** — 是否需要更新文档模板？

### 复盘复用的产出

复盘阶段结束时，应该有：
1. ✅ 复用清单（哪些东西可以复用？如何使用？）
2. ✅ 流程改进建议（哪里可以优化？如何优化？）
3. ✅ 约定更新建议（是否需要更新 CLAUDE.md / AGENTS.md？）

### 复盘复用的频率

**不是每个任务都需要正式的复盘**，但应该保持这个意识：
- 小任务：脑子里过一遍就行
- 中等任务：在 log.md 末尾写几句
- 大任务：写正式的 review.md

## 其他指导原则

### 1. 需求驱动

所有工作必须围绕需求展开。没有需求支撑的工作不做。

**检查清单：**
- [ ] 这个工作对应哪个需求？（R-xxx）
- [ ] 这个需求经过确认了吗？
- [ ] 有明确的任务拆解吗？（T-xxx）

### 2. 实时记录

每个操作都必须记录到 log.md，包括：
- 做了什么
- 为什么这么做
- 遇到了什么问题
- 如何解决的

**不要做事后总结** — 记录必须是实时的。

### 3. 透明复盘

每次任务完成后写 review.md，包括：
- 做得好的地方
- 做得不好的地方
- AI 犯的错误
- 改进建议

**不要美化** — 诚实记录失败和弯路。

### 4. 用户优先

**用户的时间比 AI 的时间更宝贵。**

- 不要在用户没要求的情况下做额外工作
- 不要在用户没确认的情况下做重大决策
- 不要隐藏问题 — 遇到问题立即告知

### 5. 持续改进

每次任务都是学习机会：
- 从失败中学习（记录踩坑）
- 从成功中学习（总结最佳实践）
- 从用户反馈中学习（调整工作方式）

## 工作流示例

### 场景：用户说"加一个登录功能"

#### ❌ 错误做法（10-0-90-0）

```
用户：加一个登录功能
AI：好的，我开始写代码...
（90% 时间写代码，没有调研，没有验证设计，没有复盘）
AI：完成了！
用户：这不是我想要的...
```

#### ✅ 正确做法（70-20-8-2）

```
用户：加一个登录功能

AI（调研阶段 - 70%）：
- 理解意图：用户想要什么？（用户名密码？第三方登录？）
- 调研现状：当前有没有相关代码？（查看 entry/src/）
- 识别风险：HarmonyOS 的登录 API 怎么用？（查阅文档）
- 收集信息：参考其他 HarmonyOS 应用的登录实现
- 提问澄清：
  - 需要支持哪些登录方式？
  - 需要记住密码吗？
  - 需要跳转到哪个页面？

AI（验证阶段 - 20%）：
- 定义完成标准：
  - 用户可以通过用户名密码登录
  - 登录成功后跳转到首页
  - 登录失败显示错误提示
  - 支持"记住密码"选项
- 设计验证方法：
  - 手动测试：输入正确/错误的凭据，验证跳转和提示
  - 自动化测试：编写 UI 测试用例
  - 截图证据：登录成功/失败的界面截图
- 规划验收步骤：用户手动测试，确认功能符合预期

AI（实施阶段 - 10%）：
- 执行实现：
  - 创建 LoginPage
  - 实现登录逻辑
  - 集成用户认证 API
- 记录过程：实时写入 log.md，包括踩坑
- 验证结果：按照验证方案测试
- 交付成果：提交代码，通知用户验收

AI：完成了！请验收。
[附上验证证据：截图、测试结果]
```

## 总结

**70-20-8-2 法则的本质：**

> 不要急于动手。先搞清楚要做什么（70%），再想清楚怎么证明做对了（20%），然后动手做（8%），最后想一想怎么改进（2%）。

这是 AI 协作的核心方法论，也是 kouol 项目的指导原则。

遵循这个原则，我们可以：
- 减少返工（70% 调研）
- 确保价值（20% 验证）
- 高质量交付（8% 实施）
- 持续改进（2% 复盘）

**记住：慢就是快。而持续改进，让快变得更快。**
